## 一、製品概要

### 1.1 何をするのか

訪問者が **3分で** 自社に最適な AI 導入プランの「提案見積書」を受け取れる。

- **入口**: nebulainfinity.com 上の Quiz フォーム（3フェーズ・9問）
- **出力**: PDF 提案見積書（4-6ページ）をメールで送付
- **中身**: 現状診断 + 3プラン提案（松竹梅）+ 概算見積もり（モジュール分解型）+ 導入ロードマップ（3段階モデル）

### 1.2 なぜ Phase 0 か

| 判断軸 | 説明 |
|--------|------|
| コスト | ~52h 開発、運営コストほぼゼロ（$0.30-0.50/レポート + Quiz LLM ~$0.001/回）|
| リスク | 極小。失敗してもサイトにフォームが残るだけ |
| Lead 獲得 | メールアドレス必須 → 確実にリード獲得 |
| ブランディング | 「AI で即時提案見積書を出せる会社」→ 競合との差別化 |
| 検証目的 | 月間 Lead 数で Phase 1（Agent Demo）への投資判断ができる |

### 1.3 ターゲットユーザー

- **プライマリ**: 中小企業の経営者・事業部長（非技術者）
- 「AI で何かできないか」と漠然と考えているが、何から始めればいいかわからない
- 見積もり依頼のハードルが高い（「問い合わせたら営業がしつこそう」）
- → **匿名で提案見積書が得られる** のが最大の価値

## 三、Quiz 設計

### 3.1 Quiz 全体構成

```
Phase 1: 基本情報（固定 5問、~1分）
  Q1.   AI 活用の方向性
  Q1.5  新規構築 or 既存システム改善
  Q2.   具体的な内容（自由記述）
  Q3.   業種 + 従業員規模

      ↓ LLM 解析（Cloudflare Worker 内 sonnet 呼出、~2秒）

Phase 2: 業務詳細（動的 3-4問、方向固定・選択肢AI生成、~1.5分）
  Q4.  現在の方法/ツール
  Q5.  必要な機能（複数選択）
  Q6.  処理ボリューム
  Q7.  連携先システム

Phase 3: 実務情報（固定 3問、~30秒）
  Q8.  運用方法/プラットフォーム（Q1 の回答で選択肢変化）
  Q9.  導入時期
  Q10. メールアドレス + 送信

合計: 10-11問、所要時間 3-4分
```

### 3.2 Phase 1: 基本情報（固定）

**Q1. どのような AI 活用をお考えですか？**
```
○ 既存業務の自動化・効率化
  例：経費精算、受発注管理、在庫管理の自動化

○ AI を組み込んだ新しいツール・機能の開発
  例：文書分析ツール、需要予測システム、チャットボット

○ AI アシスタント・エージェントの導入
  例：営業アシスタント、カスタマーサポートBot、社内ナレッジBot
```

**Q1.5 これは新しいシステムの構築ですか？**
```
○ 新規構築（ゼロから開発）
○ 既存システムへの AI 機能追加・改善
```

**Q2. 具体的にお聞かせください（自由記述）**
```
例：「契約書のレビューを効率化したい。月100件くらいの契約書を3人でチェックしている」
例：「既存のECサイトに商品レコメンド機能を追加したい」
```

**Q3. 業種と従業員規模を教えてください**
```
業種: [選択] IT・通信 / 製造 / 小売・EC / 金融 / 不動産 / 医療・ヘルスケア / 教育 / 物流 / 飲食・サービス / その他

従業員規模: [選択] 1-10名 / 11-50名 / 51-100名 / 101-300名 / 301-1000名 / 1000名以上
```

### 3.3 Phase 2: 業務詳細（動的）

Phase 1 完了後、Cloudflare Worker が sonnet を呼び出し Q2 を解析。3-4問の選択肢を動的生成。

**重要：質問の方向（何を聞くか）は固定。LLM が生成するのは選択肢の内容のみ。**

| # | 固定の方向 | LLM 動的生成部分 |
|---|-----------|----------------|
| Q4 | 「現在、{domain}をどのように行っていますか？」 | {domain}名、4-5個の選択肢（業界特化ツール名等） |
| Q5 | 「特に必要な機能を選んでください（複数選択可）」 | 6-8個の機能候補（業務領域に特化） |
| Q6 | 「現在の処理ボリュームは？」 | 適切な単位と4段階の選択肢 |
| Q7 | 「連携が必要な既存システムはありますか？（複数選択可）」 | 業界でよく使われるシステム名 |

※ Q1.5 = 「既存改善」の場合、Q4 に「既存システムの技術情報」も含める

**LLM 呼出仕様:**
- 呼出元: Cloudflare Worker（Vela ではない）
- モデル: sonnet（高速・低コスト）
- 入力: ~200 tokens、出力: ~300 tokens（JSON）
- レスポンス: ~1-2秒
- 出力形式: 厳格な JSON schema
- フォールバック: LLM 失敗時は場景別のデフォルト選択肢を使用

**JSON schema example:**
```json
{
  "domain": "契約書レビュー",
  "q4": {
    "subject": "契約書レビュー",
    "options": ["LegalForce等の専用ツール", "Word+メールでのやりとり", "法務部門が手作業で確認", "外部法律事務所に委託", "その他"]
  },
  "q5": {
    "options": ["条項リスク自動判定", "修正案の自動生成", "契約書テンプレート管理", "多言語対応", "期限管理・アラート", "バージョン比較", "承認ワークフロー"]
  },
  "q6": {
    "unit": "月間の契約書処理件数",
    "options": ["〜30件", "30〜100件", "100〜300件", "300件以上"]
  },
  "q7": {
    "options": ["Slack/Teams", "社内文書管理システム", "電子契約サービス(CloudSign等)", "CRM/SFA", "なし/わからない"]
  }
}
```

### 3.4 Phase 3: 実務情報（固定）

**Q8. 運用方法/プラットフォーム**

Q1 の回答に応じて選択肢が変わる:

**Q1 = 自動化ワークフローの場合:**
```
ワークフローの運用方法を選んでください
○ 完全自動（スケジュール/イベント実行、UI 不要）
○ 簡易管理画面あり（実行状況の確認・手動起動）
○ チャットツール連携（Slack/LINE/Teams から操作・通知）
○ 既存業務ツールへの組み込み
```

**Q1 = AI エージェントの場合:**
```
AI エージェントの主な利用シーンを選んでください
○ 顧客対応（Web チャット、LINE 等での問い合わせ対応）
○ 社内業務支援（Slack/Teams Bot、社内ポータル）
○ 自律実行型（データ収集・レポート、結果通知のみ）
```

**Q1 = AI アプリの場合:**
```
アプリケーションのプラットフォームを選んでください
○ Web アプリ
○ iOS アプリ
○ Android アプリ
○ クロスプラットフォーム（iOS + Android）
○ 未定
```

→ Q8 の回答が見積の UI/UX モジュール構成を直接決定する

**Q9. 導入希望時期はいつ頃ですか？**

### 4.4 第2章：提案（3プラン）

**AI に生成させる内容：**
- プラン A / B / C の概要・フロー・期待効果・リスク
- 各プランの機能モジュール構成
- 各プランの推奨理由

**3プランの位置づけ：**

| プラン | 名称 | AI化率 | 対応 Stage |
|--------|------|--------|-----------|
| Plan A | フル自動化 | ~95% | Stage 1→2→3 |
| Plan B | サポート型 | ~40% | Stage 1 のみ |
| Plan C ★推奨 | ハイブリッド | ~75% | Stage 1→2 |

**Prompt 指示：**
```
第2章「提案」を書いてください。

1. Q2 自由記述と Q5 必要機能から、業務を機能モジュールに分解
2. 各モジュールの複雑度（4軸）と実現難易度（4軸）を評価
3. ランク（簡単/中等/複雑/要検討）を判定
4. プラン A/B/C それぞれに含めるモジュールを決定：
   - Plan A: 全モジュール
   - Plan B: コアモジュールのみ（簡単〜中等ランク中心）
   - Plan C: コア + 主要拡張モジュール（★推奨）

- 含まれる機能モジュール一覧（ランク付き）
- 業務フロー（AI/人間の担当を明示）
- 期待効果（定量的な KPI：削減率、時間、コスト）
- メリットとリスク（各2-3項目）

| 既存改善 | 統合度 | PoC/パイロット | 本格統合 | 最適化 |

### 5.1 コスト構造（3層）

| ランク | 条件 | 単価 |
|--------|------|------|

**C層：インフラ・ランニング費用（月額）**

- AI API 利用料（算出基準: $1/100万トークン(入力+出力)。処理量に応じた概算）
- クラウドインフラ（コンピュート、ストレージ、ネットワーク）
- 保守・サポート = **初期費用（A層+B層）× 20% ÷ 12**（年間保守費の月額換算）

### 5.2 機能モジュール評価基準

#### 複雑度（4軸で評価）

| 評価軸 | 低 | 中 | 高 |
|--------|-----|-----|-----|
| データ処理 | 単純な CRUD | フォーマット変換、集計 | AI推論、NLP、画像認識 |
| ロジック | 線形フロー | 条件分岐、ルールエンジン | 状態遷移、複合判定、学習 |
| 入出力 | 単一フォーマット | 複数フォーマット | リアルタイム、双方向同期 |
| 精度要件 | ベストエフォート | 高精度(95%+)、人間確認あり | ミッションクリティカル(99%+) |

**判定ルール**: 4軸の最頻値。2軸以上が「高」→ 複雑度「高」

#### 実現難易度（4軸で評価）

| 評価軸 | 低 | 中 | 高 |
|--------|-----|-----|-----|
| 外部依存 | なし or 標準ライブラリ | メジャーAPI(Anthropic,AWS等) | 特殊API、レガシー連携 |
| 技術成熟度 | 実績豊富 | 安定版あるが事例少 | R&D要素、PoC必要 |
| 環境要件 | 標準クラウド(CPU) | GPU/大容量ストレージ | オンプレ、専用HW、閉域網 |
| 規制・セキュリティ | 基本認証・暗号化 | 業界基準(ISMS等) | 法規制(APPI厳格,医療,金融) |

**判定ルール**: **1軸でも「高」→ 実現難易度「高」→ 要具体検討**

#### 複雑度 × 実現難易度 → ランク・単価マトリクス

```
                  実現難易度
                低         中         高
複雑度  低    ¥40,000    ¥40,000    要検討
        中    ¥40,000    ¥60,000    要検討
        高    ¥60,000    ¥80,000    要検討
```

### 5.2.1 機能分解の必須カテゴリ

4. インフラ（認証、監視、セキュリティ、デプロイ）

### 5.2.2 文書処理レベル判定

企業の私有文書を扱うモジュールは、文書処理の複雑さによってコストが大幅に異なる。
以下の3段階で判定し、L3 の場合は追加モジュールが必要となる。

| レベル | 条件 | 例 | 追加モジュール |
|--------|------|-----|--------------|
| L1: 固定文書 | 固定パス・少量・構造化済み | マニュアル、FAQ、テンプレート | なし |
| L2: 構造化検索 | DB/API 経由のデータ取得 | CRM データ、注文情報、在庫 | なし |
| L3: RAG/セマンティック検索 | 大量の非構造化文書 | 社内ナレッジ、過去の報告書、雑多な資料 | **3モジュール追加** |

**L3 判定時に追加されるモジュール：**

| モジュール | カテゴリ | 内容 | ランク | 単価 |
|-----------|---------|------|--------|------|
| ドキュメント前処理パイプライン | データ管理 | 既存文書のクレンジング・チャンク分割・メタデータ付与。文書量・品質により複雑〜要検討 | 複雑〜要検討 | ¥80,000〜別途 |
| ベクトル DB 構築・管理 | インフラ | embedding 生成、インデックス構築、更新パイプライン | 中等 | ¥60,000 |
| セマンティック検索エンジン | AI コア | クエリ処理、リランキング、コンテキスト組立 | 中等 | ¥60,000 |

**C層への追加影響（L3 の場合）：**
- embedding API 月額（文書量に比例、目安: ~1万文書で ¥5,000-10,000/月）
- ベクトル DB ホスティング（Pinecone/Qdrant/Weaviate 等、¥3,000-15,000/月）

**判定ロジック（Quiz 回答から）：**
- Q2 に「大量の既存文書」「紙の資料」「社内ナレッジ」「過去データ」→ L3
- Q5 に「文書検索」「ナレッジ活用」「類似文書検索」→ L3
- Q4 が「ファイルサーバー」「共有フォルダ」「紙ベース」→ L3（前処理が重い）
- Q4/Q5 が「FAQ」「マニュアル」「テンプレート」「固定帳票」→ L1-L2

### 5.2.3 実装方式の判定（Skill 型 vs Workflow 型）

**自動化ワークフロー**シナリオでは、実装方式によってコスト構造が根本的に異なる。

| 比較軸 | Skill 型（AI Agent + Skill 定義） | Workflow 型（n8n/Dify 等） |
|--------|--------------------------------|--------------------------|
| インフラ | 最小限（AI API のみ） | 専用サーバー、ワークフロー環境 |
| 開発速度 | 速い（Prompt + ツール定義） | 遅い（ノード設計・接続・テスト） |
| 精度 | LLM 依存（~95%、例外あり） | 高精度（決定的実行パス、99%+） |
| 保守コスト | 低（Prompt 更新で対応） | 高（ノード管理・バージョン管理） |
| 拡張性 | 高（Skill 追加で即拡張） | 中（ワークフロー再設計が必要） |
| 適性 | 柔軟な業務、変化が多い | 定型業務、高精度必須、大量処理 |

**コスト構造の違い：**

| 項目 | Skill 型 | Workflow 型 |
|------|---------|------------|
| RD（設計） | **重い**（Prompt 設計が核心） | 標準的 |
| Dev（実装） | 軽い（コード量少） | **重い**（ノード設定・テスト多い） |
| B層 単価 | 標準（既存ランク表適用） | **高め**（環境構築モジュール追加） |
| C層 月額 | AI API のみ | AI API + **ワークフロー基盤**（¥10,000-30,000/月） |

**3プランへの統合：**

| プラン | 実装方式 | 特徴 |
|--------|---------|------|
| Plan B（梅） | Skill 型・最小構成 | 低コスト・速い・精度 ~95%・将来拡張可 |
| Plan C（竹）★推奨 | Skill 型・充実構成 | バランス型・十分な精度・段階拡張可 |
| Plan A（松） | Workflow 型 or Skill+Workflow 混合 | 高精度・高コスト・本格運用向け |

> Plan C 推奨理由：Skill 型で十分な精度が出せる範囲で最適化。将来精度要件が上がれば Plan A（Workflow 型）にアップグレード可能。段階移行は「追加投資」であり「やり直し」ではない。

**Plan A（Workflow 型）の追加モジュール：**

| モジュール | カテゴリ | 内容 | ランク | 単価 |
|-----------|---------|------|--------|------|
| ワークフロー環境構築 | インフラ | n8n/Dify 等のサーバー構築・設定 | 中等 | ¥60,000 |
| ワークフロー設計・実装 | データ管理 | ノード設計、接続、エラーハンドリング | 複雑 | ¥80,000 |
| ワークフロー監視・ログ | インフラ | 実行ログ、エラー通知、リトライ設定 | 簡単 | ¥40,000 |

**判定ロジック（Quiz 回答から）：**

```
自動化シナリオの場合:
  IF Q6 処理量 = 大（300件+/月）
     AND 業務が定型的（経理・発注・在庫管理等）
    → Plan A に Workflow 型を推奨
  ELIF Q6 = 小〜中
     OR 業務が柔軟/変化が多い
    → Plan B/C に Skill 型を推奨
  ELSE
    → Plan B/C = Skill 型、Plan A = Workflow 型 として両方提示
```

**注意：この分類は自動化ワークフローシナリオにのみ適用。**
AI エージェント・AI アプリ・既存改善シナリオは実装方式の分岐なし。

- {TIMELINE} を考慮したスケジュール調整
- {SCALE} で期間を調整（小規模: ×0.7、大規模: ×1.3）
```

### 6.3 Prompt のトークン数見積もり

### 7.2 技術スタック

| 要素 | 技術 | 理由 |
|------|------|------|
| Quiz UI | Next.js 静的導出 | 既存サイトと統合 |
| API / バリデーション | Cloudflare Worker | 無料枠、エッジ実行 |
| Quiz 動的生成 | Cloudflare Worker + sonnet | Phase 2 質問を 1-2 秒で生成 |
| Rate Limit | Cloudflare KV | IP ベース、1日3回制限 |
| Data Store | Cloudflare D1 (SQLite) | 永続化、リトライ可能 |
| AI（レポート生成） | OpenClaw Agent "Vela" (opus) | 柔軟、コード変更なしで調整可能 |
| PDF 生成 | Puppeteer on NAS | 完全制御、日本語フォント対応 |
| メール送信 | Microsoft Graph API (Outlook) | 既存 Outlook サーバー + サービストークン、専用メールアドレス |
| 予約システム | Microsoft Graph API (Outlook) | 既存インフラ活用、追加コストなし |
| Lead 記録 | Cloudflare D1 | 同一 DB |

### 7.3 アーキテクチャ詳細

**フロントエンド → Worker（Phase 2 動的生成・同期）:**
1. ユーザーが Phase 1（Q1-Q3）を回答
2. Worker が sonnet を呼び出し（~200 tokens in, ~300 tokens out）
3. Q4-Q7 の質問・選択肢を JSON で返却（1-2秒）
4. フォールバック：LLM 失敗時はシナリオ別デフォルト質問を返却
5. ユーザーが Phase 2（Q4-Q7）+ Phase 3（Q8-Q9）を回答

**フロントエンド → Worker（全回答送信・同期）:**
1. ユーザーが全回答を送信
2. Worker がバリデーション + レート制限チェック（KV: IP ベース 3/日）
3. 使い捨てメールブラックリスト照合
4. 同一メール頻度チェック（D1: 月3回まで）
5. D1 に quiz submission を書き込み（status="pending"、全9問の回答を保存）
6. 即座に「受付完了」レスポンスを返却

**Vela Agent → D1（非同期 cron）:**
1. Vela（OpenClaw Agent, opus モデル）が cron で D1 を定期ポーリング
2. status="pending" のエントリを取得
3. Quiz 回答（9問）から Prompt を構築
4. モジュール分解 → 複雑度・難易度評価 → ランク決定 → 見積算出
5. 提案見積書（Markdown）を生成
6. HTML テンプレート + 生成コンテンツ → Puppeteer（NAS 上）で PDF を生成
7. Microsoft Graph API (Outlook) で PDF 添付メールを送信
8. D1 の status を "completed" に更新
9. タスク失敗時はデータが D1 に残り、次回 cron でリトライ

### 7.4 データフロー

```
1. ユーザーが Phase 1 回答（Q1-Q3）
   POST /api/ai-assessment/generate-questions
   Body: { scenario, freeDescription, industry, companySize }

2. Worker: sonnet で Phase 2 質問を動的生成
   → { q4, q5, q6, q7 } (JSON)
   → 失敗時: シナリオ別フォールバック質問を返却

3. ユーザーが Phase 2-3 回答（Q4-Q9）して全回答送信
   POST /api/ai-assessment/submit
   Body: { scenario, freeDescription, industry, companySize,
           currentTools, requiredFeatures, scale, integrations,
           timeline, email }

4. Worker: バリデーション
   - 入力形式チェック（全9問 + メール）
   - メール形式バリデーション
   - 使い捨てメールブラックリスト照合
   - IP レート制限チェック（KV: assessment:{IP} → count, 24h TTL）
   - 同一メール頻度チェック（D1: 月3回まで）

5. Worker: D1 に書き込み
   INSERT INTO submissions (email, scenario, free_description,
     industry, company_size, current_tools, required_features,
     scale, integrations, timeline, status, created_at)
   status = "pending"

6. Worker: レスポンス返却
   → { status: "ok", message: "提案見積書の作成を受け付けました" }

7. Vela Agent (cron): D1 ポーリング
   SELECT * FROM submissions WHERE status = "pending"

8. Vela Agent: Prompt 構築 + AI 生成
   - 全9問の回答を Prompt 変数にマッピング
   - モジュール分解 → 評価 → 見積算出 → ロードマップ生成
   - opus モデルで提案見積書（Markdown）を生成

9. Vela Agent: PDF 生成
   HTML テンプレート + Markdown → Puppeteer → PDF

10. Vela Agent: メール送信
    Microsoft Graph API (Outlook) で PDF 添付メール送信

11. Vela Agent: ステータス更新
    UPDATE submissions SET status = "completed" WHERE id = ?
```

**予約フロー（CTA 経由）：**

```
12. ユーザーが PDF CTA または /booking にアクセス
    GET /api/booking/slots?date=2026-02-20
    
13. Worker: Graph API で空き時間を取得
    GET /me/calendar/getSchedule
    → 営業時間内の空きスロットを返却

    POST /api/booking/reserve
    Body: { date, time, name, company, email }

15. Worker: バリデーション + 重複チェック（D1）

16. Worker: Graph API でイベント作成
    POST /me/events
    → Outlook カレンダーにイベント追加
    → 確認メール自動送信（Outlook 標準）

17. Worker: D1 に予約記録
    INSERT INTO bookings (name, company, email, datetime, status)
    → Lead 情報として Quiz submissions と統合管理
```
